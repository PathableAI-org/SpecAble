## Release goal

SpecAble v0 delivers a local-first, open-source CLI that validates product
primitive fixture graphs against the canonical Product Primitives ontology,
reports relationship integrity issues, and generates deterministic Markdown
summaries and JSON reports — fully offline via `specable check`, with no Notion,
MCP, or cloud runtime dependencies.

## Release thesis

Product intent scattered across prose documents degrades quickly. A typed
primitive graph with status-aware validation (`Draft` / `Active` / `Deprecated`)
lets engineers, product owners, and coding agents share durable product state
locally before any write-back or hosted tooling exists. v0 proves that
offline validation, integrity reporting, and human-readable summaries are
useful on their own.

## Target users / usage mode

- **Local developers and contributors** maintaining primitive fixture graphs in a
  repository and running `specable check` in CI or pre-commit workflows.
- **Product owners and engineers** modeling objectives, capabilities, workflows,
  and stories as structured JSON fixtures using bundled synthetic examples.
- **Coding agents** consuming validation JSON, integrity reports, and summaries
  as machine- and human-readable artifacts without live service dependencies.

All examples use synthetic data only (generic and CoachBridge-inspired fixtures).

## Scope

- Canonical nine-type Product Primitives ontology (Objective, Actor, Persona,
  Domain Concept, Capability, Capability Concept Link, Expected Result, Workflow,
  Story)
- Local JSON fixture loading and graph indexing
- Status-aware structural validation with Draft warnings vs Active failures
- Relationship integrity reporting (orphans, duplicate names, story triples,
  workflow derivability, and related advisories)
- Deterministic Markdown summary and JSON report generation via `specable check`
- CLI exit codes: `0` (pass), `1` (Active failures), `2` (usage/decode errors)
- Bundled example graphs: generic and CoachBridge-inspired (synthetic only)
- Publishable `@specable/cli` package with Changesets release workflow

## Out of scope

- Notion, Confluence, Linear, Jira, GitHub, or Figma write-back
- MCP server or hosted graph runtime
- Product Experience Context, Design Impact Review, or design artifact handoff
- Roadmap generation, vertical slice planning automation, or task generation
  beyond Spec Kit in downstream milestones
- YAML fixture format (v0 uses JSON exclusively)

## Success criteria

- [ ] A user can run `specable check <dir>` on a local JSON fixture folder with
  no network access and receive validation status, integrity findings, and a
  summary preview on stdout
- [ ] Active validation failures and broken references cause exit code `1`;
  integrity warnings alone do not fail the command
- [ ] Fixture decode and usage errors cause exit code `2` with actionable messages
- [ ] `--out <dir>` writes `summary.md`, `validation.json`, and
  `integrity-report.json` deterministically for the same input graph
- [ ] Bundled generic and CoachBridge-inspired synthetic examples validate and
  demonstrate correct, incorrect, Draft, Active, and Deprecated handling
- [ ] `@specable/cli` publishes via Changesets with documented install and usage

## Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Ontology rules diverge from canonical Notion definition | Encode rules in typed schemas and tests; document clarifications in Spec Kit spec |
| Scope creep into hosted/MCP features | Release and milestone out-of-scope sections; constitution gates |
| Fixture format churn | JSON-only v0 with explicit schema decode tests at boundaries |
